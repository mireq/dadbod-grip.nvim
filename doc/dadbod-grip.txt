1. INTRODUCTION                                          *grip-introduction*

dadbod-grip.nvim is a database client for Neovim with support for
PostgreSQL, MySQL, SQLite, DuckDB, and MotherDuck. Each engine gets
schema browsing, inline cell editing, and metadata inspection.

Renders query results as editable grids with visual change staging,
sort/filter/pagination, foreign key navigation, schema browsing, and a
SQL query pad. Works standalone or alongside vim-dadbod-ui.

Quick start: >
  :GripConnect                  pick or add a database connection
  :GripSchema                   browse tables with columns and types
  :Grip users                   open a table in an editable grid
  :GripQuery                    open a SQL scratch pad
<

2. REQUIREMENTS                                          *grip-requirements*

- Neovim 0.10 or later

  PostgreSQL ......... psql
  SQLite ............. sqlite3
  MySQL/MariaDB ...... mysql (8.0.3+ for --csv, or MariaDB 10.5+)
  DuckDB ............. duckdb

3. INSTALLATION                                              *grip-install*

lazy.nvim (recommended): >lua
  {
    "joryeugene/dadbod-grip.nvim",
    version = "*",
    cmd = { "Grip", "GripSchema", "GripTables", "GripQuery", "GripConnect",
            "GripStart", "GripHome" },
    keys = {
      { "<leader>lg", function() require("dadbod-grip").open_smart() end,
        desc = "Grip: Open grid" },
      { "<leader>gs", "<cmd>GripSchema<cr>", desc = "Grip: Schema browser" },
      { "<leader>gt", "<cmd>GripTables<cr>", desc = "Grip: Table picker" },
      { "<leader>gq", "<cmd>GripQuery<cr>", desc = "Grip: Query pad" },
      { "<leader>gc", "<cmd>GripConnect<cr>", desc = "Grip: Connect" },
    },
    opts = {},
  }
<
SQL completion (nvim-cmp) is built in. No additional plugins required.
To verify your setup after installation: >
  :checkhealth dadbod-grip
<
packer.nvim: >lua
  use {
    "joryeugene/dadbod-grip.nvim",
  }
<

4. CONFIGURATION                                              *grip-config*

setup() is called automatically by the plugin loader with sensible defaults.
Override if needed: >lua
  require("dadbod-grip").setup({
    limit         = 100,    -- default row limit for SELECT queries
    max_col_width = 40,     -- max display width per column
    timeout       = 10000,  -- query timeout in milliseconds
  })
<
AI SQL generation (optional): >lua
  require("dadbod-grip").setup({
    ai = {
      provider = nil,       -- nil = auto-detect, or "anthropic"/"openai"/"gemini"/"ollama"
      model = nil,          -- nil = provider default
      api_key = nil,        -- nil = env var, "env:VAR", "cmd:op read ...", or direct string
      base_url = nil,       -- override for ollama or proxy
    }
  })
<
Provider auto-detection priority: ANTHROPIC_API_KEY > OPENAI_API_KEY >
GEMINI_API_KEY > ollama (local). Explicit provider setting always wins.

No default keymaps are set outside the grip buffer.

5. COMMANDS                                                 *grip-commands*

All commands are global and available after the plugin loads.

                                                                     *:Grip*
:Grip [table|SQL|file|URL]
  Open a table in an editable grid, run a SQL query, open a local file,
  or query a remote file via DuckDB httpfs.

  Examples: >
    :Grip users
    :Grip SELECT * FROM orders LIMIT 50
    :Grip /path/to/data.parquet
    :Grip https://example.com/data.csv
<

                                                               *:GripSchema*
:GripSchema
  Open or focus the schema browser sidebar. Shows tables and views with
  expandable columns, type abbreviations, and PK/FK markers. The `gb`
  keymap focuses the sidebar from any buffer, or closes it when already
  inside.

                                                               *:GripTables*
:GripTables
  Open the table picker. Shows a floating list with column preview (PK,
  FK markers, and types). Uses grip_picker with zero external dependencies.
  Also reachable via `go`, `gT`, or `gt` from any grip buffer.

                                                                *:GripQuery*
:GripQuery [sql]
  Open a SQL query pad. Optionally pre-fill with a SQL string. Press
  <C-CR> to execute the buffer contents into a grip grid.

                                                                 *:GripSave*
:GripSave [name]
  Save the current query pad content to .grip/queries/{name}.sql.
  Prompts for a name if not provided.

                                                                 *:GripLoad*
:GripLoad [name]
  Load a saved query into the query pad. Shows a picker if no name is
  provided.

                                                              *:GripConnect*
:GripConnect [url]
  Switch database connection. Shows a picker if no argument is provided.
  Reads from .grip/connections.json, g:dbs (DBUI compatible), and
  $DATABASE_URL.

                                                              *:GripExplain*
:GripExplain [sql]
  Query Doctor: translates EXPLAIN plans into plain-English health
  checks. Shows severity (SLOW/WARN/OK), proportional cost bars,
  and actionable suggestions (e.g. "add an index on the filtered
  column"). Works across PostgreSQL, MySQL, DuckDB, and SQLite.

                                                          *:GripProperties*
:GripProperties [table]
  Show full table properties in a floating window. Includes columns with
  types, primary keys, foreign keys, indexes, row estimate, and table size.
  Uses the current grip table if no argument is provided.

                                                              *:GripProfile*
:GripProfile [table]
  Open a data profiling report showing per-column statistics with Unicode
  sparkline distributions. Shows completeness, cardinality, min/max, and
  top values. Adapts to narrow terminals with a stacked layout.

                                                                 *:GripAsk*
:GripAsk [question]
  Generate SQL from a natural language question using AI. Schema context
  is auto-assembled from database metadata. Result appears in the query
  pad for review before execution.

  Providers: Anthropic, OpenAI, Gemini, Ollama (local).
  Auto-detects from env vars: ANTHROPIC_API_KEY > OPENAI_API_KEY >
  GEMINI_API_KEY > ollama fallback. Override with setup({ai={...}}).

                                                              *:GripHistory*
:GripHistory
  Browse query history. Every query executed via :Grip, the query pad,
  and staged DML is recorded in .grip/history.jsonl. Shows a floating
  picker with timestamp and SQL preview. Selecting an entry opens it in
  the query pad for review and re-execution.

                                                                *:GripDiff*
:GripDiff {table1} {table2}
  Compare two tables row-by-row using primary keys. Shows changed cells,
  added rows, and deleted rows with color-coded diff highlighting.
  Auto-selects compact mode on narrow terminals (<120 columns).
  Navigate changes with ]c and [c. Toggle compact/wide with gv.

                                                              *:GripRename*
:GripRename {old_name} [new_name]
  Rename a column in the current table. If new_name is provided, executes
  immediately. If only old_name is provided, prompts for the new name.

                                                              *:GripCreate*
:GripCreate
  Open an interactive table designer. Prompts for table name and columns,
  previews the CREATE TABLE DDL, and asks for confirmation before executing.

                                                                *:GripDrop*
:GripDrop [table]
  Drop a table with typed confirmation (must type table name to confirm).
  Detects foreign key dependents and offers CASCADE if they exist.

                                                              *:GripAttach*
:GripAttach [dsn] [alias]
  Attach an external database to the current DuckDB session. The attached
  database becomes queryable via its alias in SQL (e.g. pg.users,
  local.orders). Extensions (postgres_scanner, mysql_scanner,
  sqlite_scanner, motherduck) are installed and loaded automatically.

  If no arguments are provided, prompts for DSN and alias interactively.

  Examples: >
    :GripAttach postgres:dbname=sales host=localhost user=me  sales
    :GripAttach sqlite:legacy.db  legacy
    :GripAttach mysql:host=db.internal user=reader database=shop  shop
    :GripAttach md:shared_analytics  cloud
<
  Attachments persist in .grip/connections.json under the parent connection
  and are restored on reconnect.

                                                              *:GripDetach*
:GripDetach [alias]
  Detach a previously attached database. Removes it from the session and
  from persisted connection metadata. The schema sidebar updates to remove
  the database section.

                                                               *:GripOpen*
:GripOpen [path]
  Open any data source directly without saving to connections. Accepts
  file paths, HTTPS URLs, and s3:// paths (via DuckDB httpfs).
  If no argument is provided, opens the connection picker (same as gc).

  Examples: >
    :GripOpen ~/data/report.parquet
    :GripOpen https://example.com/dataset.parquet
    :GripOpen s3://my-bucket/data.parquet
    :GripOpen s3://my-bucket/prefix/
<
  For s3:// prefixes (no file extension), opens the query pad pre-filled
  with a glob() query. For specific files, opens an editable grid directly.
  Requires DuckDB as the active connection for file and S3 access.

                                                               *:GripStart*
:GripStart
  Open the built-in demo database (Softrear Inc. Analyst Portal). Creates
  a pre-seeded SQLite database in the plugin data directory and opens the
  schema browser. A fast way to try every feature without a real database.

                                                                *:GripHome*
:GripHome
  Return to the dadbod-grip welcome screen from anywhere. Shows the keymap
  reference, connection string examples, and navigation shortcuts. Also
  reachable with Q from any grip buffer.

7. EDITING                                          *grip-editing-workflow*

Grip uses a staging model for edits. Changes are staged in memory and
color-coded in the grid before being applied to the database.

Color coding:
  Violet ........... Modified cell (staged change)
  Red .............. Row marked for deletion
  Green ............ Newly inserted row

Workflow:
  1. Edit cells with `i` or `<CR>`, set NULL with `x`, paste with `p`
  2. Insert rows with `o`, mark deletions with `d`
  3. Preview staged SQL with `gs` or live preview with `gl`
  4. Apply all changes with `a` (wraps in BEGIN/COMMIT transaction)
  5. Undo individual edits with `u` or reset all with `U`

All DML is wrapped in a transaction. If any statement fails, the entire
transaction is rolled back and your staged changes are preserved so you
can fix the value and retry.

Multi-level undo: every edit operation pushes the previous state onto a
50-deep undo stack. Press `u` repeatedly to step back through your edit
history. Press `U` to reset to the original state in one step.

Read-only mode is auto-detected when no primary key exists on the table
(grip needs PKs for WHERE clauses in UPDATE/DELETE statements).

8. SORT, FILTER, PAGINATE                           *grip-sort-filter-page*

Sort ~
Press `s` on a column header to cycle through ASC, DESC, and off. The
query is re-run with an ORDER BY clause. Press `S` to stack a secondary
sort (multi-column sorting).

Filter ~
Press `f` on a cell to quick-filter by that value (column = value).
Press `gn` to filter by IS NULL on the column under the cursor.
Press `gF` to open the filter builder: choose an operator (=, !=, >,
<, LIKE, IN, IS NULL, IS NOT NULL) and enter a value. Numeric strings
are passed unquoted; all others are single-quoted with escaping.
Press `<C-f>` to enter a freeform WHERE clause. Multiple filters are
AND-ed together. Press `F` to clear all filters.

Filter presets ~
Press `gP` to save the current active filters as a named preset for the
current table. Presets are stored in `.grip/filters.json` keyed by table
name. Press `gp` to load a saved preset using the grip_picker float.

Paginate ~
Results are paginated (default 100 rows per page). The status line shows
the current page and total. Press `H`/`L` to navigate pages (or `]p`/`[p`),
or `]P`/`[P` to jump to the last/first page.

Press `X` to reset all sort, filter, and pagination state at once.

9. FOREIGN KEY NAVIGATION                               *grip-fk-nav-full*

Press `gf` on a cell that contains a foreign key value to navigate to the
referenced row in the target table. A new grid opens showing just that
row. Press `<C-o>` to go back. The title bar shows a breadcrumb trail
of your navigation path.

FK metadata is fetched automatically via information_schema (PostgreSQL,
MySQL, DuckDB) or PRAGMA foreign_key_list (SQLite).

11. QUERY PAD                                             *grip-query-pad*

  <C-CR>          Execute buffer contents into a grip grid (normal/insert)

13. CONNECTIONS                                          *grip-connections*

`:GripConnect` manages database connections from multiple sources:

  .grip/connections.json     project-local connection profiles
  g:dbs                      vim-dadbod-ui compatible table
  $DATABASE_URL              environment variable
  g:db / b:db                vim-dadbod variables

Add a new connection: >
  :GripConnect postgresql://user:pass@localhost/mydb
<
Or run `:GripConnect` with no arguments to pick from known connections
or add a new one interactively.

Connections are deduplicated by URL across all sources.

14. EXPORT                                                   *grip-export*

Press `gE` in a grid to export the current view to the clipboard. Six
formats are available:

  CSV .............. RFC 4180 compliant, quoted fields
  TSV .............. Tab-separated values
  JSON ............. Array of objects
  SQL INSERT ....... INSERT statements for the current table
  Markdown ......... GitHub-flavored markdown table
  Grip Table ....... Box-drawing table matching the grid style

All formats are copied to the system clipboard.

Export to file ~                                              *:GripExport*
Press `gX` or run `:GripExport` to write the current result set to a
file. You are prompted for a format (csv, json, or sql) and a file path
(defaults to the current directory). NULL values are represented as
empty fields (CSV), JSON `null`, or the SQL `NULL` keyword. The table
name is used in SQL INSERT statements; raw query results use
`_grip_result` as the table name.

Additional yank commands:
  y ................ Yank single cell value
  Y ................ Yank entire row as CSV
  gY ............... Yank entire table as CSV

15. SCHEMA OPERATIONS                                            *grip-ddl*

Table properties ~
Press `gI` in a grid or run `:GripProperties` to open a floating window
showing full table metadata: columns with types, primary keys, foreign
keys, indexes, estimated row count, and table size.

Properties view keymaps:
  R               Rename column under cursor
  +               Add new column (prompts for name, type, default)
  x               Drop column under cursor (typed confirmation)
  gI              Refresh properties
  q / <Esc>       Close

Column rename ~
Press `R` in the properties view or run `:GripRename old_name new_name`.
Previews the ALTER TABLE DDL and asks for confirmation before executing.

Column add/drop ~
Press `+` in properties to add a column. Prompts for column name, data
type, and optional default value. Previews the DDL before executing.
Press `-` in properties to drop a column. Requires typing the column
name to confirm (destructive operation).

Create table ~
Run `:GripCreate` or press `+` in the schema browser. Opens an
interactive column designer that prompts for table name, then each
column (name, type, primary key). Previews the CREATE TABLE DDL and
asks for confirmation.

Drop table ~
Run `:GripDrop` or press `D` in the schema browser. Checks for foreign
key dependents and offers CASCADE if they exist. Requires typing the
table name to confirm.

16. ADAPTERS                                              *grip-adapters*

Grip supports four database adapters. The adapter is selected automatically
based on the connection URL scheme.

  postgresql:// .... PostgreSQL via psql CLI
  sqlite: .......... SQLite via sqlite3 CLI
  mysql:// ......... MySQL/MariaDB via mysql CLI
  duckdb: .......... DuckDB via duckdb CLI

Each adapter implements: query, execute, get_primary_keys, get_column_info,
get_foreign_keys, get_indexes, get_table_stats, list_tables, and explain.

File-as-table: `:Grip /path/to/data.parquet` routes through DuckDB to
query Parquet, CSV, JSON, NDJSON, and XLSX files directly. No database
connection is needed for file queries.

Remote file querying: `:Grip https://example.com/data.csv` opens remote
files via DuckDB's httpfs extension. The extension is loaded automatically.
HTTP queries use a 30-second timeout. Supported formats are the same as
file-as-table. All remote queries open in read-only mode.

DuckDB federation: attach external databases via :GripAttach. Supports
PostgreSQL (postgres_scanner), MySQL (mysql_scanner), SQLite
(sqlite_scanner), and MotherDuck. See |grip-federation| for details.

17. PUBLIC API                                                  *grip-api*

                                                          *grip.setup()*
grip.setup({opts})
  Optional configuration override. Called automatically with defaults by
  the plugin loader.

  Parameters:
    limit         (number)  Default row limit. Default: 100
    max_col_width (number)  Max display width per column. Default: 40
    timeout       (number)  Query timeout in ms. Default: 10000

                                                           *grip.open()*
grip.open({arg}, {url}, {opts})
  Open a table, run a query, or open a file.

  Parameters:
    arg   (string)  Table name, SQL query, or file path
    url   (string)  Connection URL (optional, uses current connection)
    opts  (table)   Optional: { reuse_win = winid }

                                                     *grip.open_smart()*
grip.open_smart()
  Auto-detects context from the current buffer:
    - DBUI SQL buffer: opens that table, reuses dbout window
    - dbout result buffer: traces back to source table
    - Normal buffer: uses word under cursor as table name

18. QUERY HISTORY                                            *grip-history*

Every query executed through :Grip, the query pad, and staged DML is
recorded in .grip/history.jsonl. Each entry includes the SQL, timestamp,
connection URL, table name, query type, and execution time.

Browse history with `:GripHistory` or `gh` from a grid buffer. The floating
picker shows timestamp, elapsed ms, and SQL preview. Selecting an entry
opens it in the query pad for review and re-execution.

History is stored per-project (relative to .grip/ directory) and grows
as an append-only JSONL file.

19. DATA DIFF                                                    *grip-diff*

Compare two tables row-by-row using primary keys with `:GripDiff`.

  :GripDiff users users_backup

The diff view shows:
  - Changed cells highlighted in yellow
  - Added rows highlighted in green
  - Deleted rows highlighted in red
  - Unchanged rows (hidden by default, toggle with `u`)

Diff keymaps:
  ]c / [c         Next / previous change
  gv              Toggle compact / wide layout
  u               Toggle unchanged rows
  q / <Esc>       Close diff

Auto-selects compact mode on narrow terminals (<120 columns). Compact
mode shows changes in a stacked format; wide mode shows side-by-side.

20. TABLE PROFILING                                          *grip-profiling*

Profile table columns with `:GripProfile` or `gR` from a grid buffer.

Shows per-column statistics:
  - Unicode sparkline distributions (block characters)
  - Completeness percentage (non-NULL ratio)
  - Cardinality (distinct value count)
  - Min/max values for numeric and date columns
  - Top values with frequency counts

Adapts layout to terminal width. Narrow terminals get a stacked layout;
wide terminals get a tabular layout.

Profiling keymaps:
  q / <Esc>       Close profile float

21. QUERY DOCTOR                                        *grip-explain-full*

The Query Doctor translates raw EXPLAIN plans into plain-English health
checks. Run with `:GripExplain` or `gx` from a grid buffer.

Output includes:
  - Severity labels: OK (green), WARN (yellow), SLOW (red)
  - Proportional cost bars showing relative expense of each operation
  - Actionable tips (e.g. "add an index on the filtered column")
  - Summary with estimated total cost and row count

Severity rules adapt to context:
  - Sequential scans on small tables (<1000 rows) downgrade to OK
  - Nested loops with few rows downgrade to OK
  - Filters with low cost (<500) downgrade to OK

Works across all four adapters:
  - PostgreSQL: full cost and actual timing from EXPLAIN ANALYZE
  - MySQL: cost and row estimates
  - DuckDB: estimated cardinality
  - SQLite: EXPLAIN QUERY PLAN operations

22. AI SQL GENERATION                                             *grip-ai*

Generate SQL from natural language with `:GripAsk` or `gA` from a grid.

  :GripAsk show me the top 5 users by order count

Schema context is auto-assembled from database metadata (table names,
column names, types, primary keys, foreign keys). The generated SQL
appears in the query pad for review before execution.

Supported providers:
  - Anthropic (Claude) via ANTHROPIC_API_KEY
  - OpenAI (GPT) via OPENAI_API_KEY
  - Google Gemini via GEMINI_API_KEY
  - Ollama (local) as automatic fallback

Auto-detection priority: Anthropic > OpenAI > Gemini > Ollama.
Override with setup({ai={provider="openai"}}).

API key resolution: env var name, "env:CUSTOM_VAR", "cmd:op read ...",
or a direct string. See |grip-config| for the full ai config block.

23. CROSS-DB FEDERATION                                    *grip-federation*

DuckDB can attach external databases and query across them in a single
session. Grip exposes this through :GripAttach and :GripDetach commands.

Attach a PostgreSQL database: >
  :GripAttach postgres:dbname=sales host=localhost user=me  pg
<
Attach a local SQLite file: >
  :GripAttach sqlite:legacy.db  legacy
<
Query across both plus the local DuckDB: >
  :Grip SELECT p.name, l.total
    FROM pg.customers p
    JOIN legacy.orders l ON p.id = l.customer_id
    WHERE l.total > 1000
<
The schema sidebar groups tables by database when attachments are active:

  main (12 tables)
  pg (8 tables)
  legacy (3 tables)

MotherDuck (cloud DuckDB) works as both a primary connection and an
attachment: >
  :GripAttach md:shared_analytics  cloud
<
Extensions are managed automatically. Attaching a postgres: DSN installs
and loads postgres_scanner. Attaching sqlite: loads sqlite_scanner.
Attaching md: loads motherduck.

Attachments persist in .grip/connections.json under the parent DuckDB
connection. They restore automatically when you reconnect.

Sidebar keymaps for federation (DuckDB connections only):
  ga              Attach external database (prompts for DSN and alias)
  gd              Detach attached database (prompts for alias)

24. S3 AND REMOTE FILES                                       *grip-s3-files*

DuckDB can query remote files directly via the httpfs extension.

HTTPS files (no credentials needed): >
  :GripOpen https://example.com/data.parquet
<

S3 files (with credentials in environment): >
  :GripOpen s3://my-bucket/data/report.parquet
<

S3 prefix (browse bucket contents): >
  :GripOpen s3://my-bucket/data/
<
  Opens the query pad pre-filled with a glob() query.

Save a remote file as a named connection: >
  gc → + New connection → paste URL → give it a name
<

Cross-federation with remote files: >
  SELECT l.user_id, r.event_date
  FROM local.users l
  JOIN read_parquet('s3://my-bucket/events.parquet') r ON l.user_id = r.user_id
<

For S3 access, set AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY in your
environment. Public buckets work without credentials.
DuckDB's httpfs extension installs automatically on first use.

25. HEALTH CHECK                                              *grip-health*

Run the built-in health checker to verify your installation: >
  :checkhealth dadbod-grip
<

Checks performed:

  Neovim version     Confirms Neovim >= 0.10 is installed.

  Database CLIs      Checks psql, sqlite3, mysql, and duckdb are in PATH.
                     Reports WARN for any missing (only the adapters you use
                     are required).

  AI providers       Checks ANTHROPIC_API_KEY, OPENAI_API_KEY, GEMINI_API_KEY
                     environment variables and ollama executable.
                     Reports WARN if none are set (AI features will be
                     disabled but everything else works normally).

Example output: >
  dadbod-grip
    OK  Neovim >= 0.10
    OK  psql found
    OK  sqlite3 found
    WARN  mysql not found (required for that adapter)
    OK  duckdb found
    OK  Anthropic API key set (ANTHROPIC_API_KEY)
<
